SECCIÓN 1

Etapas Reales de un Proyecto MLOps

1️⃣ Definición del Problema

Identificar el objetivo de negocio, definir métricas de éxito y establecer los criterios de aceptación del modelo.

2️⃣ Recolección y Exploración

Obtener datos, realizar EDA, validar calidad, detectar sesgos y definir la estrategia de features.

3️⃣ Preprocesamiento

Limpieza, imputación, encoding, escalado. Todo dentro de Pipelines para evitar data leakage.

4️⃣ Entrenamiento y Validación

Comparar modelos con validación cruzada, registrar experimentos en MLFlow, seleccionar el mejor.

5️⃣ Empaquetado y Despliegue

Serializar modelo, contenerizar con Docker, exponer como API REST, CI/CD automatizado.

6️⃣ Monitoreo y Reentrenamiento

Detectar data drift, logging de predicciones, reentrenamiento programado o por degradación.

4 Pipeline de Preprocesamiento

Usamos ColumnTransformer de Scikit-Learn para evitar data leakage.

# Variables Numéricas
num_transformer = Pipeline([
  ('imputer', SimpleImputer(
      strategy='median')),
  ('scaler', StandardScaler())
])

# Variables Categóricas
cat_transformer = Pipeline([
  ('imputer', SimpleImputer(
      strategy='most_frequent')),
  ('onehot', OneHotEncoder(
      handle_unknown='ignore'))
])

¿Qué hace cada paso?

SimpleImputer: Rellena valores nulos (mediana para números, moda para texto).

StandardScaler: Normaliza a media=0 y desviación=1.

OneHotEncoder: Convierte categorías en columnas binarias (0/1).

SECCIÓN 4

⚠️ ¿Qué es el Data Leakage?

El data leakage ocurre cuando información del conjunto de test se filtra al proceso de entrenamiento, generando métricas optimistas pero falsas.

❌ Incorrecto (con leakage)

1. Escalar todo el dataset
2. Luego dividir en train/test
3. El scaler aprendió estadísticas del test

Resultado: El modelo "ve" datos del futuro. Las métricas no reflejan rendimiento real.

✅ Correcto (sin leakage)

1. Dividir primero en train/test
2. Crear Pipeline con preprocesamiento
3. pipeline.fit(X_train) solo toca train

Resultado: El ColumnTransformer ajusta solo con train y transforma test de forma independiente.

Por eso usamos Pipeline + ColumnTransformer: garantizan que el fit() solo ocurre en entrenamiento.

5 Tracking con MLFlow

¿Por qué registrar experimentos?

Cuando evaluamos múltiples modelos, necesitamos comparar métricas, versionar modelos y reproducir resultados. MLFlow automatiza todo esto.

mlflow.set_experiment("California_Housing_Prediction")

with mlflow.start_run(run_name=model_name):
    pipeline.fit(X_train, y_train)
    predictions = pipeline.predict(X_test)

    rmse = np.sqrt(mean_squared_error(y_test, predictions))
    r2 = r2_score(y_test, predictions)

    mlflow.log_metric("rmse", rmse)
    mlflow.log_metric("r2", r2)
    mlflow.sklearn.log_model(pipeline, "model")

📏 ¿Cómo Interpretar las Métricas?

📐 RMSE — Root Mean Squared Error

Penaliza más los errores grandes. Si un modelo predice $200K pero el valor real es $500K, ese error pesa mucho más que varios errores pequeños.

Interpretación: "En promedio, el modelo se equivoca por ~$48,077" (ponderando errores grandes).

📊 MAE — Mean Absolute Error

Error promedio simple. Más fácil de interpretar porque no amplifica outliers. Cada error pesa igual.

Interpretación: "En promedio, la predicción se desvía ~$31,989 del valor real".

📈 R² — Coeficiente de Determinación

% de varianza explicada. R²=0.82 significa que el modelo explica el 82% de la variabilidad del precio.

Rango: 0 (no explica nada) → 1 (predicción perfecta). Puede ser negativo si el modelo es peor que la media.

🔄 CV RMSE σ — Estabilidad del Modelo

Desviación estándar entre folds. Una σ baja indica que el modelo es consistente y generaliza bien.

Señal de alerta: Si σ es alta, el modelo es inestable — su rendimiento depende mucho de qué datos se usen.

VALIDACIÓN CRUZADA

🔄 ¿Qué es la Validación Cruzada (Cross-Validation)?

La validación cruzada K-Fold divide los datos de entrenamiento en K particiones. Se entrena K veces, cada vez usando una partición diferente como validación y el resto como entrenamiento. Esto da una estimación más robusta del rendimiento real del modelo.

Fold 1 → Val
|
Fold 2 → Val
|
Fold 3 → Val
|
Fold 4 → Val
|
Fold 5 → Val

✅ Ventajas

• Usa todos los datos para entrenar y validar
• Reduce el riesgo de overfitting al test set
• Detecta inestabilidad del modelo (alta varianza entre folds)

⚙️ Implementación

from sklearn.model_selection import cross_val_score

cv_scores = cross_val_score(
    pipeline, X_train, y_train,
    cv=5, scoring='neg_mean_squared_error'
)
cv_rmse = np.sqrt(-cv_scores)

VALIDACIÓN CRUZADA

Logging de CV en MLFlow

# Cross-Validation (5-fold) en cada modelo
cv_scores_neg_mse = cross_val_score(
    pipeline, X_train, y_train,
    cv=5, scoring='neg_mean_squared_error', n_jobs=-1
)
cv_rmse_scores = np.sqrt(-cv_scores_neg_mse)

# Registrar métricas de CV en MLFlow
mlflow.log_metric("cv_rmse_mean", cv_rmse_scores.mean())
mlflow.log_metric("cv_rmse_std", cv_rmse_scores.std())
for i, fold_rmse in enumerate(cv_rmse_scores):
    mlflow.log_metric(f"cv_rmse_fold_{i+1}", fold_rmse)

💡 ¿Por qué registrar CV en MLFlow?

• Comparar estabilidad entre modelos
• Detectar overfitting (test RMSE ≪ CV RMSE)
• Trazabilidad completa del proceso de evaluación

📈 Métricas adicionales registradas

• cv_rmse_mean: RMSE promedio de 5 folds
• cv_rmse_std: Desviación entre folds
• cv_rmse_fold_1..5: RMSE individual por fold

6 Almacenamiento en Parquet

¿Por qué Parquet y no CSV?

• Columnar: Lee solo las columnas necesarias
• Compresión: 2-10x menos espacio
• Tipos nativos: Preserva tipos de datos
• Ecosistema: Spark, BigQuery, Athena

import pyarrow as pa
import pyarrow.parquet as pq

# Guardar predicciones
test_results = X_test.copy()
test_results['actual'] = y_test
test_results['predicted'] = (
    best_model.predict(X_test)
)

table = pa.Table.from_pandas(test_results)
pq.write_table(table,
    'test_predictions.parquet')
